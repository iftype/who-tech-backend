리팩토링 목적은 기술 부채 해소보다 **AI 협업 비용 절감**이 우선이다.

**Why:** 파일이 크고 복잡하면 수정 시 읽어야 할 컨텍스트가 많아져 토큰이 과도하게 소비됨. 구조가 명확하고 파일이 작을수록 변경 작업당 필요한 컨텍스트가 줄어듦.

**How to apply:**

- 리팩토링 시 "파일 크기 축소 + 단일 책임 원칙" 우선
- 함수/파일을 작게 쪼개 개별 변경이 좁은 범위에서 이루어지도록
- 타입 명확화로 추론 오류 방지 → 재시도 비용 절감
- 대형 파일(sync.service.ts, members.js 등)은 우선 분리 대상
